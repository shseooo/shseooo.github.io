Local LLM에 푹 빠져지냈던 한 달

계기

요즘 AI 열풍이 무섭다. 거의 매주 새로운 기술과 모델이 발표되어, 따라가기도 버거운 시대에 살고 있다.

나의 밥줄인 개발에도 AI가 침범해왔다. 몇 년 전까지만 해도 “아직 쓸 만한 수준은 아니다”라고 생각했는데, 어느새 대격변의 시대가 온 것을 느낀다. Claude Code로 개발 업무를 해보면 가끔 바보짓도 하지만, 놀라울 따름이다.

새로운 기술들(예: OpenClaw)을 저렴한 유지비로 써보고 싶어서 새 맥북을 구입했다. 그런데 맥북 살 돈이면 AI 서비스를 몇 년은 구독할 수 있지 않나? …맞다. 사실 그냥 새 맥북이 사고 싶었다.

그리고 어떻게든 생산적인 일을 하고 싶어서, 한 달 동안 고군분투한 기록을 남긴다.

Local LLM 입문

나의 환경

  • CHIP : M5 Pro (20코어)
  • RAM : 64GB

고려해야 하는 것들

  1. Model (모델)
  2. Quantization (양자화)
  3. Context Window 사이즈
  4. Max Tokens (최대 출력 토큰 수)
  5. KV 캐시
  6. TurboQuant
  7. AI Agent
  8. AI Backend

…등등.

모델 선택 과정

가장 중요한 것은 모델 선택이다. 실행 가능한 사이즈의 모델 중 가장 퀄리티가 높은 것을 골라야 한다.

최초 후보는 Gemma4와 Qwen3.6이었다. Gemma4는 코딩에 적합하지 않다는 평이 많아서 Qwen3.6으로 시작했다.

Qwen3.6에는 35B와 27B가 있는데, 간단히 말하면 성능은 27B, 속도는 35B다. 가능하면 27B를 쓰고 싶어서 속도를 개선할 방법을 찾아 나섰다.

속도에 영향을 미치는 요소

  • 양자화 수준 — 쉽게 말하면 원본을 압축한 정도다. 메모리에 올라가는 모델의 크기를 결정한다. Qwen3.6에서 도구 호출을 문제없이 하려면 권장 양자화가 Q5 이상이라고 한다.
  • Context Window 사이즈 — 모델이 한 번에 기억할 수 있는 자료의 크기다. 메모리가 넉넉하다고 최대로 키우면 출력 속도가 느려지므로, 적당한 크기를 고르는 게 중요하다.
  • TurboQuant — Context를 압축하는 기술이다. 지능은 거의 떨어뜨리지 않으면서 메모리 사용량을 크게 줄여주고, 속도도 소폭 개선된다고 한다.
  • MTP / DFlash — 내가 이해한 바로는, 작은 모델이 예측 토큰을 생성하고 큰 모델이 적당한 토큰을 선택하는 방식으로 속도를 개선한다. 둘의 자세한 차이는 솔직히 잘 모르겠다.
  • GGUF / MLX 등 — 모델을 구동하는 방식에 따라서도 속도 차이가 있다. 맥에서는 MLX가 좋은 선택이다.

나의 선택

1. 양자화

모델 크기를 보고 골랐다. 약 30GB 모델이 적당하다고 판단했다.

64GB 중 16GB는 일반적인 사용(브라우저, Slack 등)에, 나머지는 AI에 할당하기로 했다. 모델 + 컨텍스트를 합쳐 48GB 안에 들어와야 하므로, 모델은 30GB 정도가 한계다.

  • Qwen3.6-27B (Q6) : 약 28GB
  • Qwen3.6-35B-A3B (Q6) : 약 27GB

2. Context Size

처음에는 Qwen3.6이 지원하는 최대 256k로 하고 싶었지만, 벤치마크로 측정한 응답 속도를 생각하면 64k가 적당했다.

다만 64k로 AI Agent를 돌려보면 생각보다 자주 Auto Compact이 발동해서 조금 아쉽다.

3. TurboQuant

논문이 발표된 지 몇 달 되지도 않은 신기술이다. 4bit 압축을 해도 지능 손실이 거의 없다고 한다. bit 수가 낮을수록 압축률은 높아지지만 지능 손실이 생길 수 있다.

단, MTP나 DFlash를 쓰는 경우에는 함께 사용할 수 없다.

4. MTP, DFlash

둘 다 사용해본 결과 — 둘 다 쓰지 않기로 했다…!

Qwen3.6-35B-A3B

OriginalMTPDFlash
50 tok/s100 tok/s250 tok/s

DFlash는 벤치마크에서 압도적인 결과를 얻었지만, 실제 사용에서는 Original과 별반 다르지 않았다. MTP 역시 벤치마크보다 속도가 크게 떨어졌고, 프롬프트가 한글이면 더 느려진다는 이야기도 있었다.

정리하면, 사용하지 않기로 한 이유는 다음과 같다.

  • TurboQuant를 적용할 수 없어 메모리 압박이 크다.
  • 실제 출력 성능 향상은 미미했다.
  • 지능에서도 소폭 손해가 있다.

처음 MTP가 나왔을 때는 열광했는데, 실제로 써보고는 마음이 팍 식어버렸다.

사실 MTP·DFlash는 27B를 제대로 활용하고 싶어서 많이 시도해본 것이었다.

OriginalMTPDFlash
8 tok/s25 tok/s50 tok/s

역시 벤치마크에서는 쓸 만했지만 실사용에서는 13 tok/s 수준으로 내려왔다. 27B는 너무 답답해서 결국 35B를 선택하게 되었다.

5. Backend

Backend는 모델을 구동하고 관리하는 서버다. 많이 쓰는 것으로는 Ollama, llama.cpp, oMLX, vLLM 등이 있다.

  • Ollama — 사용성은 좋지만 성능이 떨어지고 라이선스 문제가 있다.
  • llama.cpp — GGUF 형식의 모델을 구동한다. Windows·Linux 등에서도 쓸 수 있어 호환성이 좋고, CLI 기반이라 상세 설정이 가능하며 성능도 좋다. 다만 UI가 없다는 점이 특징이다.
  • oMLX — MLX 모델 전용 서버로, 맥에서 쓸 수 있는 최고의 Backend라고 생각한다. 모델 관리·벤치마크·모니터링이 모두 쓰기 쉬운 UI로 구성돼 있다.
  • vLLM — 사용해보지 않았다.

llama.cpp와 oMLX를 번갈아 써본 결과, 최종적으로 oMLX를 골랐다. 출력 토큰 속도가 5 tok/s 정도 더 빠르기도 하고, 모델 관리의 편의성이 너무 좋다. 만세!

6. Agent

모델과 설정을 마치고 일을 시키려면 적당한 하네스(agent)가 필요하다.

  • pi — “최소한의 agent”라는 컨셉으로, 가장 잘 동작한다.
  • opencode — reasoning에 토큰을 계속 소모하다가 멈춰버린다.
  • Claude Code — 현재 최신 oMLX에서 오류가 발생해 사용할 수 없다. 커스텀 모델의 컨텍스트 사이즈를 설정할 수 없어서, 작업하다 보면 컨텍스트 용량 초과로 뻗어버린다.

여러 가지를 테스트해봤는데, “딸깍” 개발을 할 때는 Claude Code가 제일 잘 동작한다. 하지만 여러 오류로 지금은 쓸 수 없다. 차선책으로 pi와 opencode를 써봤고, pi가 마음에 들었다. “필요한 기능은 AI에게 시켜서 추가한다”는 개념과 심플함이 좋았다.

요즘 유행하던 hermes agent도 써봤는데, 첫 프롬프트부터 무한 루프에 빠지는 걸 보고 바로 지워버렸다.

일단 pi와 OpenClaw로 가기로 했다.

7. Model

Huggingface를 검색해보면 무수히 많은 Qwen3.6의 변형이 존재한다. 사고방식을 Claude Opus로 바꾼 것, 새로운 양자화 방식 oQ, MTP 모델 등등.

각 모델을 양자화 단계별로 테스트하고 top_p·top_k 설정도 바꿔보며 별짓을 다 해봤지만, 결론은 근본 모델이 최고더라. unsloth 모델부터 테스트했는데, 최종적으로 unsloth의 MLX 모델이 제일 좋았다. 유명한 집은 이유가 있는 법이었다.

OpenClaw

ChatGPT가 처음 출시됐을 때만큼이나 큰 인기를 끈 앱이다. 그런데 아직 베타 버전이다.

Claude Code와 MCP를 연계하면 침대에 누워서 “딸깍” 개발이 가능해진다. 내부적으로는 pi로 동작하며, 개인 비서로 만족스럽게 움직여 준다. 쇼핑 추천, 뉴스 알림, 고민 상담, 코딩 등에 활용하고 있는데, 잘 동작해서 기특하다.

이 과정에서 터널링이라는 기술을 알게 됐는데, 신기해서 더 깊이 파보고 싶어졌다.

최종 설정 (26-06-01)

  • Backend : oMLX
  • Model : Qwen3.6-35B-A3B-6bit
  • TurboQuant : 4bit
  • Context Window : 64k
  • Max Tokens : 16k
  • Agent : pi, OpenClaw

소감

AI 개발을 해보니, 할 수 있는 영역은 엄청나게 넓어지는데 정작 내 실력은 점점 떨어지는 기분이 든다. 휴대폰이 대중화되며 전화번호를 외우지 않게 된 것처럼, 코딩 능력이 퇴화해가는 느낌이다. 그래서 non-AI 코딩을 꾸준히 하는 것도 중요한 것 같다.

이제 업무는 AI 코딩으로, 취미로 non-AI 코딩을 하는 이상한 시대가 되어가고 있다.

Local LLM은 솔직히 아직 발전의 여지가 많아 보인다. 내년에 나올 새 모델은 정말 실무에 쓸 수 있을지도 모르겠다.

이 블로그에 있는 툴과 미니게임은 모두 AI 코딩 연습의 결과물이다. 내가 직접 만들었다면 한참 걸렸을 도구들이 뚝딱 만들어지니, 재미있으면서도 무섭다. 예전 같으면 돈 주고 샀을 앱들도 이제는 “그냥 만들어서 쓸까?” 하는 생각이 먼저 드는 요즘이다.

35B 모델이라 하더라도 좋은 결과를 내는 방법을 연구해 보아야겠다.
